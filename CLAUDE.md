# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

This project is managed with **uv** and targets **Python 3.14**.

```bash
uv sync                              # install dependencies into .venv
uv run crawler.py                    # full crawl (metadata + MD5 hashing)
uv run crawler.py --no-hash          # fast pass: metadata only, no hashing
uv run crawler.py --hash-only        # only hash rows already in DB that have md5 IS NULL
uv run crawler.py --hash-dupes       # only hash size-collision groups (skip unique-size files; the fast dedup path)
uv run crawler.py --prune            # remove rows for files deleted from disk (skips offline volumes)
uv run crawler.py --prune-excluded   # remove rows now covered by the exclude list (retroactive exclude; compact after)
uv run crawler.py --reindex-changed  # refresh rows whose file changed (size/mtime); --no-hash to skip rehashing
uv run crawler.py --roots / /Volumes/NAME  # selective scan: restrict the walk to these volume roots (default: all of CRAWL_ROOTS)
uv run compact_db.py SRC DST         # rewrite a DB into a fresh file, reclaiming dead space (used by the UI's Compact DB button)
uv run query_app.py                  # local web UI for querying the DB (http://127.0.0.1:8800)
uv run query_app.py --port 9000      # serve the UI on a different port
uv run query_app.py --host 0.0.0.0   # bind a different interface (default 127.0.0.1; keep loopback)
```

There is no test suite, linter config, or build step. The Python entry point is `crawler.py`; the query UI is `query_app.py`.

There is also an **Electron desktop wrapper** (`electron/`, `package.json`, `npm` deps) that packages the app as `Kendex.app`. It is a thin convenience layer: it picks a free port, spawns `uv run query_app.py --host 127.0.0.1 --port <port>`, and loads that loopback URL in a Chromium window — so it serves the *exact same page* as the browser. The browser-tab/LaunchAgent workflow needs none of it; `npm`/Electron only matter when building the `.app`. See the Electron wrapper section below.

External runtime dependency: the `pyexiftool` package shells out to the `exiftool` CLI binary, which must be installed on the host (`brew install exiftool`). The crawler locates the binary robustly — it tries `shutil.which("exiftool")` then the standard Homebrew locations (`/opt/homebrew/bin/exiftool`, `/usr/local/bin/exiftool`) and passes the resolved path to `ExifToolHelper(executable=…)`, so EXIF works even under a minimal `PATH` (e.g. a LaunchAgent/Electron-spawned server). If the import or the binary lookup fails, `EXIFTOOL_AVAILABLE` is false and the crawler degrades gracefully and skips EXIF extraction.

## Architecture

This is a single-host filesystem indexer for a macOS workstation with multiple mounted volumes. It walks the filesystem, records rich metadata for every file into a DuckDB database, and is designed to resume after interruption.

### Database location and self-exclusion

The DuckDB file lives at `/Volumes/TB5_DOCK8/KMSDB_PROJ/files.db` — the default in `crawler.py`, overridable per-machine via the `FILE_INDEXER_DB` env var (`query_app.py` derives its path from `crawler.DB_PATH`, so it follows; the LaunchAgent installer `install.sh` at the repo root sets the env var). The Electron build instead points the backend at a database under the app's `userData` (Application Support) dir. At startup the crawler excludes the **directory** `DB_PATH` lives in (`DB_PATH.resolve().parent`, i.e. `KMSDB_PROJ`) — not the whole volume — so the rest of that volume still gets indexed while `files.db`, its `.wal`, and the web UI's working copy (`files.db.scan`, all in the same directory) stay out of the index. During a UI maintenance run `DB_PATH` is the `.scan` copy, whose parent is the same directory, so the exclusion holds either way. If you change `DB_PATH`, the self-exclusion adapts automatically (`query_app.py` follows because it derives its path from `crawler.DB_PATH`). (The exclusion match is component-aware — exact path or `dir + "/"` prefix — so `KMSDB_PROJ` never shadows a sibling like `KMSDB_PROJ_BACKUP`.)

`CRAWL_ROOTS` is just `["/"]`. `os.walk("/")` descends into `/Volumes` (mount points are real directories), so a single `/` root already covers every mounted volume. Listing `/Volumes` as a second root would walk every external volume a **second** time, redoing MIME/MD5/EXIF on each file (the `UNIQUE` constraint stops duplicate rows, not the wasted work).

**Timestamps are stored as naive-UTC.** `ts_to_dt`/`now_utc` produce `datetime`s with `tzinfo` stripped after converting to UTC, because DuckDB's `TIMESTAMP` has no time zone and binding a tz-**aware** datetime makes the driver silently convert it to local time before storing — which reads back shifted by the UTC offset and made `--reindex-changed` treat **every** file as modified (re-hashing the whole corpus each run). Storing naive-UTC round-trips exactly. Timestamps are **stored** UTC but **displayed local**: the query UI converts at the display edge — `_jsonable` renders every `datetime` as `v.replace(tzinfo=utc).astimezone()` (the server's local zone, DST-aware), and the Locate created-date filter wraps the column in `timezone('America/Los_Angeles', timezone('UTC', created_at))` so "created June 8" means the local day. Caveat: hand-written SQL that *filters* on a raw timestamp column still compares in UTC (only display and the Locate filter are localized). `ts_to_dt` also catches `OverflowError`/`OSError`/`ValueError` and returns `None` (stored NULL) for a corrupt/sentinel stat timestamp out of the platform `time_t` range — otherwise that `OverflowError` (not an `OSError`) escapes the per-file handlers and aborts the whole `os.walk` mid-crawl.

### Resume semantics

The `files.path` column has a `UNIQUE` constraint, and inserts use `INSERT OR IGNORE`. At the start of a full crawl the crawler loads every already-indexed path into an in-memory Python set (`indexed_paths`, one `SELECT path FROM files`), and before doing any expensive work (MIME detection, MD5, EXIF) it skips files whose path is in that set — an O(1) membership test rather than a per-file `SELECT` round-trip to DuckDB (which matters because DuckDB is columnar/OLAP, so point lookups are comparatively slow). The `UNIQUE` column + `INSERT OR IGNORE` remain the real dedup backstop; the set is purely an optimization, and it's a snapshot taken at crawl start (safe because `os.walk` yields each path once per run). This means re-running `crawler.py` after a Ctrl-C resumes from where it left off; it does **not** re-scan or refresh existing rows. To force a re-index of a path, delete its row first, then re-run the crawler.

The `--hash-only` mode is a separate code path that fills in MD5 for rows where `md5 IS NULL`, ordered by `size_bytes` ascending so small files complete first.

The `--hash-dupes` mode is `--hash-only` narrowed to **size-collision groups** — it hashes only `md5 IS NULL` rows whose `size_bytes` is shared by another file (`size_bytes IN (SELECT size_bytes FROM files WHERE size_bytes > 0 AND coalesce(is_symlink,false)=false GROUP BY size_bytes HAVING count(*) > 1)`, symlinks/zero-byte excluded to match the full crawl). Rationale: two files can only be byte-identical if they're the same size, so hashing a **unique-size** file reads every byte just to confirm it has no possible twin — wasted I/O. Same size is *necessary but not sufficient*, so collision-group members still get a real MD5 to confirm. This makes the **fast dedup workflow** `--no-hash` (metadata pass) → `--hash-dupes` (hash only what could be a duplicate), typically far faster than hashing the whole corpus. Unique-size files keep `md5 IS NULL`, which is why the **Duplicate files (by MD5)** preset filters `WHERE md5 IS NOT NULL` — it stays correct, since every file that *could* be a duplicate has been hashed. (The full crawl still hashes everything by default; making metadata-first the default, or exposing `--hash-dupes` as a UI button alongside a metadata-only scan, is a deliberate follow-up left to the front-end.) The implementation shares the `--hash-only` loop via a `dupes_only` flag on `crawl()`; dispatched as `--hash-dupes`.

Because the crawl never refreshes existing rows, the DB also drifts out of sync when files are **deleted** from disk (the row lingers). The `--prune` mode is a separate code path that closes that gap: it walks every indexed `path`, checks it with `os.path.lexists`, and deletes rows whose file is gone. Critically, it only prunes a path if its volume is currently mounted (derived from the path: `/Volumes/NAME` → checked with `os.path.ismount`, everything else → `/`). Rows on unmounted/ejected volumes are skipped and reported, so running `--prune` with an external drive disconnected can never wipe that drive's index.

The `--prune-excluded` mode is the retroactive counterpart to the crawl-time exclusion: after editing the exclude list, it deletes rows whose path is now covered by `EXCLUDE_PATHS`, so an exclude edit takes effect without a from-scratch recrawl (the crawl itself only ever *avoids adding* excluded paths; it never removes already-indexed ones). It is pure path-string matching using the **same rules** as `should_skip` — plain paths are component-aware prefixes (`path == ex or path.startswith(ex + "/")`, run in DuckDB as `path = ? OR starts_with(path, ?)`) and glob entries match the whole path (`*`/`?` → DuckDB `path LIKE ? ESCAPE '\'` via `_glob_to_like`) — so unlike `--prune`/`--reindex-changed` it needs **no disk access and no mount guard** — an excluded path is removed whether or not its volume is mounted. It prints a per-entry row count for the **user** exclude list (`load_user_excludes()`) before deleting, then does one bulk `DELETE` over the full `EXCLUDE_PATHS` (defaults + user). DuckDB never returns freed blocks to the OS, so the intended workflow is **Edit exclude list → `--prune-excluded` → Compact DB** (all three are Maintenance buttons in the web UI). Exposed in the UI as the `prune_excluded` run-mode / **Prune excluded** button.

The `--reindex-changed` mode closes the remaining gap — in-place modifications. It re-stats every indexed path and refreshes the full metadata (size, MD5, MIME, stat fields, EXIF) of rows whose file **changed**, judged by size differing or mtime differing by ≥1s (second granularity absorbs sub-second float jitter in the stat-timestamp round-trip; the comparison only works because timestamps are stored **naive-UTC** — see the self-exclusion section — so `old_mtime.replace(tzinfo=utc).timestamp()` reconstructs the true epoch). It shares `--prune`'s mount guard (offline volumes skipped) and leaves missing files for `--prune` rather than deleting them. The shared mount helpers (`volume_root`, `make_mount_checker`) live just above the prune code. So a full refresh is: `--reindex-changed` (update modified) + a plain crawl (add new) + `--prune` (drop deleted).

### Schema (defined in `SCHEMA_STATEMENTS`)

- `files` — one row per file. Includes filesystem stat fields, MIME, MD5, and a flat set of `exif_*` columns plus `exif_raw` (full ExifTool output as JSON).
- `crawl_log` — one row appended per crawl invocation (start/finish/host/counters).
- `errors` — per-file errors (PermissionError, OSError) logged inline so the crawl never aborts on a single bad file.

DuckDB does not accept multiple statements per `execute()` call, so schema is a list of individual statements executed in a loop. Sequences (`seq_files`, etc.) back the auto-increment IDs because DuckDB doesn't have MySQL-style `AUTO_INCREMENT`.

### Crawl pruning

Two layers of exclusion:
1. `EXCLUDE_PATHS` — a set of exclude entries pruned from `os.walk` via `dirnames[:]` mutation before descent. Includes Time Machine snapshot mounts (`/Volumes/.timemachine`), APFS firmlink targets (`/System/Volumes/Data` — avoids double-walking), macOS system metadata (`/.Spotlight-V100`, `/.fseventsd`), and named external volumes the user wants to skip. `EXCLUDE_PATHS` is the **union** of two sets: `EXCLUDE_DEFAULTS` (hardcoded, locked — the system/TM/named-volume prefixes) and a user list loaded from `exclude_paths.json` (next to `crawler.py`) by `load_user_excludes()` at startup. **Each entry is one of two kinds** (see the exclude-matching helpers above `load_user_excludes`): a plain **absolute path** (component-aware *prefix* match — children covered automatically; `/x/FOO` never shadows `/x/FOO_BAR`), or a **glob** containing `*` (any run of chars, incl. `/`) or `?` (one char), matched against the whole path — so `*/Library/Application Support/*` excludes that folder under every user. Because a glob matches the whole path, a trailing `*` is needed to catch a folder's contents. `is_valid_exclude` accepts only absolute paths or anchored globs and **rejects match-everything patterns** (`*`, `/*`) that would otherwise let `--prune-excluded` delete the entire index; `load_user_excludes` drops anything invalid. At startup `_split_excludes` partitions the set into literal prefixes and precompiled glob regexes (`_EXCLUDE_LITERALS`/`_EXCLUDE_GLOBS`) so `should_skip` does no per-call regex compilation.
2. The DB-directory auto-exclusion described above.

The user list is edited from the web UI's **Edit exclude list** button (see Query UI section) — defaults shown read-only, user list editable, saved to `exclude_paths.json`. Editing `EXCLUDE_DEFAULTS` in source (or `CRAWL_ROOTS`) is still the way to change the locked/built-in behavior. Exclude changes apply on the **next** crawl, not retroactively.

### Selective per-volume scan (`--roots`)

`crawl()` takes an optional `roots` list (CLI `--roots PATH …`); when given, the walk is restricted to those roots instead of `CRAWL_ROOTS`. The boot disk lives at `/`, and `os.walk("/")` naturally descends into `/Volumes` (every external) — so in a **selective** scan, whenever `/` is among the chosen roots the walk prunes the `/Volumes` subtree (`prune_volumes`), and each *selected* external is reached via its own explicit `/Volumes/NAME` root. That composes correctly for any combination (boot-only, externals-only, mixed): unselected volumes are never walked, selected ones exactly once. The default full crawl (`roots=None`) is unchanged — `/` still covers everything via `/Volumes`. The web UI's **Add Files** volume picker is the front end for this (see Query UI section).

### EXIF extraction

Only files whose extension is in `EXIF_EXTENSIONS` (photo/video/audio formats) get the ExifTool pass — running ExifTool on every file would be prohibitively slow. ExifTool is invoked through a single long-lived `ExifToolHelper` context (`et_ctx`) that's reused across all files; it's torn down in the `finally` block.

### Batching

Rows accumulate in a Python list and flush to DuckDB every `COMMIT_BATCH_SIZE` (100) inserts via `executemany` + `commit`. On `KeyboardInterrupt` the current batch is flushed before exit so progress is never lost. `flush_batch` falls back to row-by-row inserts if the `executemany` raises, so one un-bindable row (e.g. a filename with bytes DuckDB can't store as TEXT) costs one row, not the whole batch — and the crawl pre-screens each filename with `path_str.encode("utf-8")`, routing non-UTF-8 names (lone surrogates from HFS+/SMB/archives, which raise a non-`OSError` that would otherwise escape the per-file handlers) to `log_error` instead of the batch.

### Progress readout

All three long loops (Indexing/Hashing/Checking) wire their `tqdm` bar to `make_readout(pbar)`, which writes a single self-overwriting line into the bar's **postfix** (so it rides the `NN%|` line the UI routes to the path pane, never scrolling the log): a file whose work took ≥1s is shown immediately as `slow Ns: <path>`, otherwise the current path is sampled at most every ~2.5s as a heartbeat so fast stretches still show roughly where the crawl is. The readout shows full absolute paths.

## Query UI (`query_app.py`)

A single-file local web UI for exploring the DB, built on stdlib `http.server` (no web framework). Binds to `127.0.0.1:8800` by default. Holds **one read-only** DuckDB connection (`_con`) serialized behind `_lock`, so queries can never mutate the index. Because the browser is a bridge from any web page to `127.0.0.1`, every request passes `_reject_if_unsafe`: the `Host` header must be `localhost`/`127.0.0.1` (defeats DNS rebinding) and POSTs must be `Content-Type: application/json` (not a CORS "simple" type, so a hostile cross-origin POST must preflight with `OPTIONS`, which the server never answers — blocking the no-preflight `text/plain` CSRF trick that could otherwise kick off a crawl or run a `SELECT` that makes DuckDB read local files and exfiltrate them via a fetched URL). Same-origin UI fetches don't preflight, so they're unaffected. Results are capped at `MAX_ROWS` (2000) so the browser never tries to render millions of rows. The page is served inline as the `PAGE` string with `__PRESETS__`/`__MAX_ROWS__` substituted via `.replace` (not `.format`, so CSS braces survive); preset reports live in the `PRESETS` list.

The **Locate files** bar (top of the main pane) is a no-SQL-required search form: a name field with a match-scope dropdown (**file name** → `filename`; **folder name** → the directory part via `regexp_replace(path, '/[^/]*$', '')`, i.e. files *inside* a matching folder, since the index has no folder rows and empty folders are invisible to it; **either** → full `path`), a **regex** checkbox (plain mode is `ILIKE '%…%'` with `%`/`_`/`\` escaped; regex mode is `regexp_matches(…, 'i')` — both case-insensitive), comma-separated extensions (normalized to the stored `.jpg` form), a created-date range (end date inclusive via `< DATE … + INTERVAL 1 DAY`), and a volume include/exclude list. It is **entirely client-side**: `locate()` builds the SQL, escapes string literals with `sqlStr()` (doubles single quotes), writes the query into the SQL textarea so the user sees/learns/can tweak it, and submits through the normal `/api/query` path — there is no separate search endpoint, and the read-only connection means a malformed query just errors. Enter in any Locate field triggers the search.

Result table headers are **click-to-sort**: `renderTable()` (the single renderer all queries flow through) wires each `<th>` to `sortBy(col)`, which sorts the already-returned rows in place — first click descending, each further click on the same column toggles direction (a ▼/▲ arrow marks the active column). `cellCmp` sorts numerically when both cells are numbers, else case-insensitive natural-string order (`localeCompare` with `numeric: true`), and nulls always sort last regardless of direction. Sorting is entirely client-side on the capped (≤`MAX_ROWS`) result set — it never re-hits the DB; running a new query resets to the DB's natural order (`sortCol = -1`).

The left sidebar (`#side`) is color-coded for quick scanning: **Presets** buttons are light blue (`#7BC9F5`), **Maintenance** buttons light red (`#F46281`), and **Clear log window** vivid yellow (`#F6DA1F`), all with near-black text and a `brightness(.9)` hover (their per-color hover rules carry extra id specificity to beat the generic gray `#side button:hover`). The **Halt & discard run** button is pulled out under its own `<hr>` and **Stop** heading at the bottom of the sidebar: it stays white but is larger (taller padding) with bold 16px Arial in strong red (`#c0182f`), so it reads as the emergency control even while `disabled` (faded via `opacity`) between runs.

Result rows highlight (light yellow) under the cursor, and hovering a row copies its path into the **path readout pane** (`#pathbox`, between the SQL box and the results table), which wraps to as many lines as needed — long paths are otherwise ellipsized in table cells. The path is found per-row as the first cell whose value looks like a path (a string starting with `/`), so it works regardless of the column's name — `path` in a plain file listing, the aliased `example` column in the Duplicate-files preset, etc. (MIME types like `image/jpeg` don't match — no leading slash). The readout keeps the last hovered path; Clear log window blanks it.

When a result carries an **`md5` column** (e.g. the Duplicate-files preset, which shows one `example` path per md5 group), each row becomes click-to-download: `dupList(md5)` runs `SELECT path FROM files WHERE md5 = … ORDER BY path` through the normal `/api/query` path, then a `confirm()` dialog reports the copy count and (after an edit-it-first warning) hands the path list to the browser as a downloaded `dupes_<md5prefix>.txt` (one plain path per line, no escaping — meant to be trimmed and fed to `tr '\n' '\0' | xargs -0 rm -f`). It's **entirely client-side** (Blob download — the read-only app never writes to disk). Because `run_query` caps at `MAX_ROWS`, a group larger than the cap is reported honestly (`N+`, with a "list is incomplete — re-run after deleting" note) rather than silently truncated.

The **Maintenance** panel shells out to the crawler so the DB can be brought current without leaving the browser, using a **copy-on-write / swap-on-success** model so a run is fully reversible:

- `start_run(mode)` spawns a background `_run_worker` thread and returns immediately. The worker first **snapshots** the live DB (`shutil.copy2(DB_PATH → WORK_DB)`, where `WORK_DB` is `files.db.scan`), then runs the crawler **against the copy** via `crawler.py --db WORK_DB`. The `COMMANDS` strings are built by `_cmd()` and always append `--db <WORK_DB>` (shell-quoted); they are fixed, never built from request input.
- Because the crawler writes to the copy, the live `files.db` is untouched during a run, so **queries stay available** (the read-only `_con` stays open). `_con` is closed only for the instant of the atomic `os.replace(WORK_DB → DB_PATH)` on success — which unlinks the old DB (not Trash) — then reopened.
- **Halt & discard** (`POST /api/run/halt`) sets `halt_requested` and SIGTERMs the crawler's **process group** (the subprocess is launched with `start_new_session=True` so `os.killpg` reaches `uv`/python/exiftool). The worker then deletes `WORK_DB` and leaves `files.db` exactly as it was. A non-zero crawler exit is treated the same as a halt (discard). A halt requested during the snapshot copy is honored before the crawler launches.
- **Compact DB** is the odd mode out: DuckDB never returns freed blocks to the OS, so `files.db` bloats across crawl/prune cycles. Instead of snapshot+crawler, the worker runs `compact_db.py`, which builds `WORK_DB` itself — `COPY FROM DATABASE` from the live DB attached read-only (preserves tables, indexes, and sequence state) — so `_run_worker` skips the snapshot step for this mode. Queries stay available, halt/discard and the swap-on-success apply unchanged.
- Run state lives in `_run` behind `_run_lock` (never nested with `_lock` — acquired sequentially to avoid deadlock), with a `phase` field (`preparing`/`running`/`halting`/`done`) the browser surfaces. `_con` is opened in `main()` via `_open_con()` (not at import) so importing the module has no side effects. The browser polls `/api/run/status` for the live log tail (`webapp_run.log`). `run_status` splits the crawler's **tqdm progress bar** out of that tail so it doesn't scroll the log: bar lines are matched by content (`_TQDM_RE` = `\d{1,3}%\||\d+file \[`, which catches them whether tqdm overwrites with `\r` or — as it does when its stream is a plain file — prints a fresh `\n` line per refresh) and returned in a separate `progress` field; the latest bar line wins. The signature covers **both** tqdm shapes: percentage bars (`40%|…`, emitted when a run has a known total — reindex/verify) **and** the counter-style line a **first scan** emits when there is no total (`917601file [3:55:20, 55.64file/s, /path]`). The counter case was added in PR #4 — before it, first-scan progress lines scrolled in the log and the path strip showed a generic "working…" during exactly the long first crawls the progress strip was built for. The browser writes `progress` into the **path readout pane** as one self-overwriting line (only when non-empty, so the final `100%` stays pinned there after the run), while the log pane keeps only the non-bar output (headers, the `===` summary block, the `[committed]`/`[discarded]` line). The **Clear log window** button is client-side only and blanks whatever pane is showing — query results/status, or the log window. Because the poller repaints the full tail every 1.5s, a mid-run clear can't simply empty the element: the click records the last 300 chars of the log as `logAnchor`, and `renderLog` thereafter shows only text after the anchor's last occurrence in the tail (anchor not found ⇒ the tail rolled past the clear point or a new run rewrote the log, so everything shown is new). Starting a run resets the anchor so its log displays from the top. Endpoints: `GET /` (page), `GET /api/run/status`, `GET /api/stats`, `GET /api/volumes`, `GET /api/excludes`, `POST /api/query`, `POST /api/open`, `POST /api/run` (accepts an optional `roots` list for Add Files), `POST /api/run/halt`, `POST /api/excludes`.

The **Add Files** button opens a **volume picker** modal (`#vpmodal`) instead of running immediately. `GET /api/volumes` (`list_volumes()`) lists the currently-mounted, non-excluded volumes — the boot disk as `/` (its `/Volumes` alias folded in as the friendly name; externals covered by the crawler exclude list are omitted) — plus `initial_scan_complete`. That flag is a JSON sidecar `files.db.state.json` (`STATE_PATH`): absent (no DB yet, or a first scan halted partway) reads false; it's written **only** when a full (all-volumes) Add Files run finishes successfully (`_scan_is_full(roots)` → `_mark_initial_complete()`). The picker defaults **all volumes checked** when the initial scan isn't complete (a first/resumable full scan) and **none checked** once it is (opt-in per scan). On **Add**, the client sends `roots: null` if every volume is checked (a clean full crawl) or the explicit selected roots otherwise; `start_run("add", roots)` validates each root against the live `list_volumes()` whitelist (rejecting anything not currently selectable) and `_add_command()` shlex-quotes them into the walk pass (`--no-hash --roots …`) while the dedup pass (`--hash-dupes`) stays **DB-wide** — so duplicates are always reported across the whole index regardless of which volumes were scanned. Request input can thus only ever pick from a server-derived set of real paths.

The **Edit exclude list** button (in the sidebar, just below the Maintenance group but deliberately *outside* `#maint` so it isn't wired as a run-mode) opens a **modal overlay** for editing the crawler's exclude list without touching source. `query_app.py` imports `crawler` as the single source of truth, so `GET /api/excludes` returns `{defaults, user}` (locked `EXCLUDE_DEFAULTS` + the `exclude_paths.json` user list) and `POST /api/excludes` validates the submitted list (`crawler.is_valid_exclude` — absolute paths or anchored globs, trimmed/deduped/sorted) and writes `exclude_paths.json` via `save_user_excludes`. The modal shows the defaults **read-only** and the user list in an editable textarea; the built-in defaults can never be removed from the UI (add-only). This is a constrained write — a JSON file of path prefixes — and does **not** feed the fixed crawler `COMMANDS`, so the run path stays free of request input. Changes apply on the next crawl; the modal says so.

The crawler's `--db PATH` flag (used only by this UI) reassigns the module-global `DB_PATH` before dispatch, so all modes — and the self-exclusion logic — operate on the given file. `WORK_DB` lives on the same volume as `DB_PATH`, so the DB-volume self-exclusion still fires and `os.replace` stays an atomic same-filesystem rename.

### Open / Reveal / Quick Look a result (`POST /api/open`)

Result rows have a context menu (and keyboard shortcut) that can **Open** a file in its default app, **Reveal** it in Finder, or **Quick Look** it (`preview`). `open_path(path, action)` runs `open <path>` / `open -R <path>` / `qlmanage -p <path>` — the path is passed as an **argv element, never through a shell** — and is the one place the read-only app *acts* on the filesystem (it still never writes the DB). It is fenced the same way as every other POST: the localhost-only `Host` check and the JSON content-type/preflight CSRF guard apply, the `action` is whitelisted to `open`/`reveal`/`preview`, the path must be absolute and **must exist on disk** (a stale/offline-volume row returns an error rather than acting), and it is macOS-only. `qlmanage` blocks while its panel is open, so `preview` spawns detached, terminates the prior panel for snappy space-bar browsing, and reaps the child in a daemon thread to avoid zombies.

### Status strip (`GET /api/stats`)

`get_stats()` returns index-level numbers for the top status strip: total row count, total bytes, per-volume row counts **with each volume's currently-mounted state** (so the UI dims rows on offline volumes), and the last successful sync time — taken from `files.db`'s mtime, which is meaningful because the copy-on-write model only replaces the file when a run commits.

### Light / dark theme

The page ships a dark "instrument-console" theme by default and a **light** theme; a `#theme-toggle` button switches between them. The choice persists in `localStorage` (`kendexTheme`) and the initial theme falls back to the OS `prefers-color-scheme`. `body.light` overrides the dark surface variables — it is pure CSS/client-side, no server involvement.

## Electron wrapper (`electron/`, `package.json`)

`electron/main.js` packages the app as `Kendex.app` for people who'd rather double-click an icon than open a browser tab. It is a thin wrapper, **not** a reimplementation:

- On launch it resolves `uv` (PATH plus the standard Homebrew / `~/.local/bin` locations), binds a free ephemeral port (`server.listen(0)`), and spawns `uv run query_app.py --host 127.0.0.1 --port <port>` as a child process, pointing `FILE_INDEXER_DB` at a `files.db` under the app's `userData` (Application Support) dir. It then `loadURL`s that loopback URL in a Chromium `BrowserWindow` — the same page the browser serves.
- During a maintenance run it holds a macOS power assertion via `powerSaveBlocker.start('prevent-app-suspension')` (≈ `caffeinate -i`, so a multi-hour scan isn't interrupted by system sleep, without forcing the display awake) and releases it when the run ends or the app quits. **This sleep-blocker exists only in the Electron wrapper** — the plain browser-tab / LaunchAgent server has no equivalent.
- On quit (`before-quit` / `window-all-closed`) it SIGTERMs the backend child so no orphaned `query_app.py` lingers.

Because the backend is unchanged `query_app.py`, anything true of the browser UI is true inside the wrapper. The two front-ends share the one `crawler.py` engine and the one `query_app.py` server; the wrapper adds only process lifecycle, a window, and the sleep assertion.

## Files to ignore when reading the repo

- `crawl.log` — runtime stderr capture from a previous run.
- `webapp_run.log` — output of the last Maintenance task launched from `query_app.py`; regenerated each run.
